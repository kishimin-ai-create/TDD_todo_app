## Summary

The implementation is well-structured and correctly handles the core CRUD flows,
soft-delete, cascade deletion, and response envelope format. Two issues stand out:
a type-coercion bug in the `completed` field update that can silently produce wrong
data, and a cascade-delete test that passes without actually exercising the cascade
path. A pair of lower-risk inconsistencies around whitespace handling and timestamp
generation are also flagged.

**<sub><sub>![P2 Badge](https://img.shields.io/badge/P2-yellow?style=flat)</sub></sub>
`Boolean()` coercion silently mis-handles non-boolean `completed` values**

**File**: `backend/src/index.ts` — lines 228-230

```typescript
if (body.completed !== undefined) {
  t.completed = Boolean(body.completed)
}
```

`Boolean()` will coerce any truthy value to `true` and any falsy value to `false`, so:

| Client sends           | Result         | Expected |
|------------------------|----------------|----------|
| `"false"` (string)     | `true` ✗       | 422      |
| `1` (number)           | `true` ✗       | 422      |
| `null`                 | `false` ✗      | 422      |
| `0` (number)           | `false` ✗      | 422      |

The spec defines `completed` as `boolean (optional)`. Sending `{ "completed": "false" }` will
mark the todo as **completed** (truthy string) instead of returning a 422 validation error.
This is a silent data-corruption bug.

**Fix**: validate that the value is strictly a boolean before applying it.

```typescript
if (body.completed !== undefined) {
  if (typeof body.completed !== 'boolean') {
    return c.json(errorRes('VALIDATION_ERROR', 'completed must be a boolean'), 422)
  }
  t.completed = body.completed
}
```

ご指摘ありがとうございます。`completed` 更新時に boolean 型を厳密に検証し、boolean 以外の値は 422 を返すよう修正しました。これで `"false"` や `1` などが誤って完了状態として保存されないようにしています。

**<sub><sub>![P2 Badge](https://img.shields.io/badge/P2-yellow?style=flat)</sub></sub>
Cascade-delete test does not actually verify cascade behavior**

**File**: `backend/src/app.test.ts` — lines 332-343

```typescript
it('200: cascade-deletes associated todos', async () => {
  const app  = await createApp('Cascade App');
  const todo = await createTodo(app.id, 'Will be cascade-deleted');

  await request('DELETE', `/api/v1/apps/${app.id}`);

  // Re-create the app so the todo route is reachable
  const app2 = await createApp('Cascade App');
  const todoRes = await request('GET', `/api/v1/apps/${app2.id}/todos/${todo.id}`);
  expect(todoRes.status).toBe(404);        // ← always 404, cascade or not
});
```

The test queries `todo.id` under `app2.id`. `findTodo` checks `t.appId === appId`, and
`todo.appId` is `app.id` ≠ `app2.id`, so the response is **always** 404 regardless of
whether cascade delete ran. The 404 proves nothing about the cascade; the test would pass
even if the cascade loop were completely removed.

**Fix**: verify the cascade by querying todos directly under the original `app.id` path *before* the
app becomes inaccessible, or by verifying that a soft-deleted todo is excluded from the list
of a reconstituted app that shares no IDs with the original. The simplest approach is to
hard-assert `todo.deletedAt` via a helper that inspects the store, or to check that the
todo list under the original app (while it still existed) would be empty if an in-process
inspection were available. A practical alternative without store access:

```typescript
it('200: cascade-deletes associated todos', async () => {
  const app  = await createApp('Cascade App');
  const todo = await createTodo(app.id, 'Will be cascade-deleted');

  // capture todo list before delete
  const beforeRes = await request('GET', `/api/v1/apps/${app.id}/todos`);
  const before = (await beforeRes.json() as any).data;
  expect(before.find((t: any) => t.id === todo.id)).toBeDefined(); // sanity

  await request('DELETE', `/api/v1/apps/${app.id}`);

  // app is gone, so re-create with a fresh ID to reach the todos route,
  // then confirm the *old* todoId is not discoverable anywhere in the store
  // by trying the ghost-app path:
  const ghostRes = await request('GET', `/api/v1/apps/${app.id}/todos`);
  expect(ghostRes.status).toBe(404); // app soft-deleted ✅

  // A cleaner proof: export an inspection helper from index.ts for tests only
});
```

Even without a store-inspection helper, the test comment should at minimum acknowledge
that the 404 is caused by a different `appId`, not cascade, so future readers don't
assume coverage that does not exist.

ご指摘ありがとうございます。ご認識の通り、現状のテストは cascade delete 自体を直接証明できていません。本件は store 状態を確認できる補助を含め、テストの検証方法を見直す対象として扱います。

**<sub><sub>![P3 Badge](https://img.shields.io/badge/P3-blue?style=flat)</sub></sub>
Whitespace is stripped for the empty check but NOT before storage or uniqueness comparison**

**File**: `backend/src/index.ts` — lines 60-65 (`validateName`) and 67-72 (`validateTitle`)

```typescript
const validateName = (name: unknown): string | null => {
  if (typeof name !== 'string') return 'name is required'
  if (name.trim().length === 0) return 'name must not be empty'  // uses trim …
  if (name.length > 100) return 'name must be at most 100 characters'  // … but length does not
  return null
}
```

And in `isNameTaken`:

```typescript
a.deletedAt === null && a.name === name  // exact byte comparison, no trim
```

Consequences:

1. `"  App  "` passes validation and is stored with surrounding spaces. A subsequent
   `POST { name: "App" }` then succeeds (409 is NOT raised), giving two live apps whose
   trimmed names are identical — effectively circumventing the uniqueness guarantee.
2. A 100-character name padded with one leading space (`" " + "a".repeat(100)`, 101 bytes)
   is rejected by the length check, even though the meaningful content is 100 characters.

**Fix**: normalize the value at the validation boundary before writing to the store:

```typescript
const validateName = (name: unknown): string | null => {
  if (typeof name !== 'string') return 'name is required'
  const trimmed = name.trim()
  if (trimmed.length === 0) return 'name must not be empty'
  if (trimmed.length > 100) return 'name must be at most 100 characters'
  return null
}
// then store `name.trim()` instead of `name`
```

Apply the same pattern to `validateTitle`.

Useful? React with 👍 / 👎.

ご指摘ありがとうございます。`name` / `title` は検証時に trim した値を基準に扱い、そのまま保存するよう修正しました。これにより前後空白で一意性や長さ判定をすり抜けることがないようにしています。

**<sub><sub>![P3 Badge](https://img.shields.io/badge/P3-blue?style=flat)</sub></sub>
`createdAt` and `updatedAt` are set by two separate `now()` calls in POST handlers**

**File**: `backend/src/index.ts` — lines 106-114 (App POST) and lines 178-188 (Todo POST)

```typescript
const record: AppRecord = {
  ...
  createdAt: now(),   // ← first call
  updatedAt: now(),   // ← second call, different Date.now() invocation
  ...
}
```

In normal execution the two timestamps are identical in practice, but they are allowed to
differ. If a future refactor slows down the path between the two calls (e.g., an `await`),
a newly created resource would report `updatedAt > createdAt` immediately upon creation,
which is semantically odd.

**Fix**: capture the timestamp once and reuse it:

```typescript
const ts = now()
const record: AppRecord = { ..., createdAt: ts, updatedAt: ts, ... }
```

Useful? React with 👍 / 👎.

ご指摘ありがとうございます。POST ハンドラではタイムスタンプを一度だけ取得し、`createdAt` と `updatedAt` に同じ値を設定するよう修正しました。生成直後に両者がずれる余地をなくしています。
