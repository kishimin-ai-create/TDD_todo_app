## 1. orval セットアップ（OpenAPI → 型 / TanStack Query / MSW 自動生成）

**コミット**: `0e32375` `feat: configure orval for type/mock/tanstack-query generation from OpenAPI spec`

### 背景

バックエンドはすでに hono-openapi で OpenAPI 定義書（`docs/spec/backend/openapi-generated-pretty.json`）を自動生成していた。フロントエンドでこれを手書きで追いかけるのは型の乖離や二重管理を生むため、定義書を単一の信頼できる情報源として、そこからフロントエンドに必要なコードを自動生成する方針を取った。

### 作業内容

- **`frontend/orval.config.ts` を新規作成**
  - `input.target` に `docs/spec/backend/openapi-generated-pretty.json` を指定
  - `output.client: 'react-query'` で TanStack Query フックを生成
  - `output.httpClient: 'fetch'` で axios 依存なしの fetch ベースに
  - `output.mock: { type: 'msw' }` で MSW ハンドラーを同時生成
  - `output.mode: 'split'` でフックは `index.ts` 一本、型定義は `models/` 以下にファイル分割
  - `override.mutator` に `src/api/client.ts#customFetch` を指定し、ベース URL 管理とエラーハンドリングを集約

- **`frontend/src/api/client.ts` を新規作成**
  - `VITE_API_BASE_URL` 環境変数でベース URL を差し込む fetch ラッパー
  - `response.ok` が false のとき JSON ボディをそのまま `throw` する

- **`frontend/package.json` にスクリプトと依存を追加**
  - `"generate": "orval"` スクリプト追加
  - `@faker-js/faker` を devDependency として追加（orval MSW 出力が内部で使用するが peerDeps に記載がなく実行時エラーになるため）

- **`frontend/vite.config.ts` に dev proxy を追加**
  - `/api` → `http://localhost:3000`（開発時の CORS 回避）

- **ESLint / TypeScript 設定の整備**
  - `eslint.config.js` の `globalIgnores` に `src/api/generated/**` を追加（生成コードへの lint 適用を除外）
  - `tsconfig.node.json` の `include` に `orval.config.ts` を追加（ESLint project service のスコープ外エラーを解消）

## 4. CI ワークフロー再設計（ci-pr.yml / ci-nightly.yml 導入・playwright.yml 廃止）

**コミット**: `c580c80` `feat: add CI workflows (PR + nightly) and update playwright config`

### 背景

既存の `playwright.yml` は以下の問題を抱えていた。

- lint・typecheck・Vitest が含まれておらず、型エラーがあっても E2E だけ走る
- 全ブラウザ × 全テストを PR のたびに実行し、最大 60 分かかる
- Storybook ビルド確認が CI になく、壊れを検知できない
- reg-suit ビジュアルリグレッションを組み込む余地がない（重すぎる）

### 作業内容

#### ci-pr.yml（PR 用・15 分タイムアウト）

- トリガー: `push` (main/develop) + `pull_request`
- ステップ構成: lint → typecheck → Vitest → Storybook ビルド → Playwright smoke（Chromium のみ・`@smoke` タグのみ）
- `--pass-with-no-tests` を付けることで、`@smoke` テストがまだ 0 本でも CI が通る
- アーティファクト保持 7 日

#### ci-nightly.yml（夜間用・60 分タイムアウト）

- トリガー: 毎日 JST 03:00（UTC 18:00）+ `workflow_dispatch`（手動起動も可能）
- 本番ビルド（`vite build`）→ `npm run preview` でサーバー起動 → Playwright 全ブラウザ・全テスト
- Storybook ビルド確認
- reg-suit ビジュアルリグレッション実行（詳細は次節）
- `wait-on` でサーバー起動を確認してから Playwright を実行
- `fetch-depth: 0` で全 git 履歴を取得（reg-keygen-git-hash-plugin の要件）
- アーティファクト保持 30 日

#### playwright.config.ts の更新

- `baseURL` を環境変数 `PLAYWRIGHT_BASE_URL` から読み込む形式に変更（フォールバックは `localhost:4173`）
- CI ではワークフロー側がサーバーを起動するため `webServer` 設定はコメントアウトのまま維持

## 5. reg-suit ビジュアルリグレッション設定（storycap 非対応問題 → Playwright 代替）

**コミット**: `7a96742` `feat: set up reg-suit visual regression with Playwright screenshots`

### 背景・問題

reg-suit の公式ドキュメントは storycap（Storybook の各ストーリーを自動スクリーンショット）との組み合わせを推奨している。しかし本プロジェクトは **Storybook 10** を使用しており、storycap は Storybook 8 以降に非対応で起動自体が失敗する。

### 解決策：Playwright でスクリーンショットを撮って `.reg/actual/` に保存する

reg-suit は「`.reg/actual/` にスクリーンショットが存在すること」だけを前提とし、取得方法は問わない。そこで Playwright で `page.screenshot()` を使い、reg-suit が期待するパスに直接保存する方式を採用した。

### 作業内容

- **`frontend/e2e/visual.spec.ts` を新規作成**
  - `test.describe('@visual', ...)` でタグ付けし、CI で `--grep "@visual"` を指定して分離実行できる
  - `page.waitForLoadState('networkidle')` で非同期コンポーネントの描画完了を待ってからキャプチャ
  - `page.screenshot({ fullPage: true })` でビューポート外のレイアウト崩れも検出
  - 保存先を `path.join(process.cwd(), '.reg', 'actual')` に明示的に指定

- **`frontend/regconfig.json` を新規作成**
  - `thresholdRate: 0.01`（1%）：サブピクセルレンダリングの差異による誤検知を防ぐ
  - `reg-keygen-git-hash-plugin`: git コミットハッシュをキーに前後の画像を紐付け
  - `reg-publish-s3-plugin`: スクリーンショットと差分レポートを S3 に保存（CI ジョブをまたいでデータ共有）
  - `reg-notify-github-plugin`（`prComment: true`）: 差分検出時に PR へ自動コメント
  - `bucketName` はプレースホルダー（CI 実行時に `jq` で GitHub Secrets から動的注入）

- **`frontend/.gitignore` に `.reg/` を追加**
  - スクリーンショット・差分レポートは S3 で管理するため git には含めない

- **インストールパッケージ**
  - `reg-suit`、`reg-keygen-git-hash-plugin`、`reg-publish-s3-plugin`、`reg-notify-github-plugin`
  - `storycap` は非対応のためインストールしていない

- **`ci-nightly.yml` への統合**
  - `@visual` テスト実行 → `jq` で `bucketName` を注入 → `npx reg-suit run` → アーティファクト保存

## 8. ReviewResponseAgent による修正対応

**コミット**: `8ac31e7` `fix: address review findings from frontend-ci-backend-20260428`

### 修正済み（5 件）

#### P1 — mysql-app-repository.ts 競合バグ修正

`save()` の SELECT + INSERT/UPDATE 2 ステップを廃止し、`mysql-todo-repository.ts` と同様の単一アトミック upsert に置き換えた。

```sql
INSERT INTO App (...) VALUES (?)
ON DUPLICATE KEY UPDATE name = VALUES(name), updatedAt = VALUES(updatedAt), deletedAt = VALUES(deletedAt)
```

`ExistsRow` 型は `existsActiveByName` で引き続き参照されるため残置。

#### P2 — ci-nightly.yml @visual 2 重実行の解消

全件実行ステップに `--grep-invert "@visual"` を付加し、`@visual` テストは専用ステップのみで 1 回だけ実行される構成に修正。

#### P2 — hono-app.test.ts テスト名修正

`'PUT with no body falls back to empty body (returns 422 for missing name)'` を `'PUT with no body is accepted as no-op update (200)'` に改名。`name` はオプショナル更新であり 200 が正しい動作であることを明示。

#### P3 — http-presenter.ts 戻り値型追加

`AppDto` / `TodoDto` を export 型として新設し、`presentApp` と `presentTodo` に明示的な戻り値型を付与。

#### P3 — ci-pr.yml @smoke テスト常駐化

`--pass-with-no-tests` を削除。`frontend/e2e/example.spec.ts` の既存テストタイトルに `@smoke` タグを追加し、CI 上でスモークテストが常に 1 件以上実行される状態を保証。
