## OrchestratorAgent のルールを強化した

### 背景

今日の TDD セッションで、OrchestratorAgent がサブエージェント（Red / Green）のファイル作成に失敗した際に、**自分でコードを書いてしまう**という問題が発生した。

- Red Agent が `app.test.ts` を作らなかった → OrchestratorAgent が general-purpose エージェント経由でテストを作成
- Green Agent が `index.ts` を実装しなかった → 同様に自前で実装

これは「オーケストレーターは指揮者であり、自分でコードを書いてはいけない」というルールに反する動作だった。

### 変更内容

`/.github/agents/OrchestratorAgent.agent.md` を v2.0.0 → **v2.1.0** に更新した。

**Prohibited Actions の強化**

```
❌ Generate code yourself — ABSOLUTE.
   Never use create/edit tools to write source code, test code, or configuration files.
   This includes recovery attempts when a sub-agent fails.

❌ Fall back to writing code when a sub-agent fails —
   retry the agent or stop and ask the user for guidance instead
```

**Workflow フェーズへの明示的な失敗時対応の追加**

- Red / Green フェーズ：エージェントがファイルを作成しなかった場合は「1回だけリトライし、それでも失敗したらユーザーに報告して止まること」を明記
- Refactor フェーズ：「変更なし」も正当な結果として受け入れること（自分でリファクタしない）

**Key Principles の強化**

```
❌ Write code — under any circumstances, including fallback
❌ Create or edit source files directly
❌ Compensate for a failing sub-agent by doing the work yourself
```

### 教訓

サブエージェントが失敗したとき、オーケストレーターが「穴を埋めるために自分でやる」のは一見合理的に見えるが、役割の分離を破壊する。失敗を見えなくするより、**失敗を報告してユーザーに判断を委ねる**方が正しい設計。

### Red フェーズ：テストを先に書く

`backend/src/app.test.ts` を新規作成した。

Hono の `app.request()` を使えばサーバーを起動せずに HTTP レベルのテストが書けるので、それを活用した。  
各テストが独立して動くように、`clearStorage()` を実装側からエクスポートして `beforeEach` でストアをリセットする設計にした。

カバレッジの設計方針：

- 正常系（201 / 200 のレスポンス形状チェック）
- バリデーションエラー（422：欠如・空文字・空白のみ・文字数超過）
- 一意制約違反（409：重複 name）
- 存在しないリソース（404）
- ソフトデリート後に一覧・単体取得ともに見えなくなること
- App 削除時に配下の Todo もカスケード削除されること
- 境界値（name が100文字 → OK、101文字 → 422）

この段階では実装がないため全テストが失敗する（Red）。

### Review フェーズ：コードレビューで2件のバグが見つかった

レビュー結果は `review/todo-api-20260425.md` に保存した。

#### バグ1（P2）：`Boolean()` 強制変換によるサイレントなデータ破壊

PUT Todo の `completed` フィールドで `Boolean(body.completed)` を使っていた。  
`"false"` という文字列を送ると `true` に変換されてしまう。

```typescript
// 修正前
t.completed = Boolean(body.completed)

// 修正後
if (typeof body.completed !== 'boolean') {
  return c.json(errorRes('VALIDATION_ERROR', 'completed must be a boolean'), 422)
}
t.completed = body.completed
```

#### バグ2（P3）：ホワイトスペースのトリムが保存まで一貫していなかった

バリデーションはトリム後の文字列で行うのに、保存はトリム前の元の値でしていた。  
`"  My App  "` と `"My App"` が別名として共存できてしまうため、一意制約が形骸化していた。

```typescript
// 修正後：保存前に trim()
const name = (body.name as string).trim()
```

両バグともレビュー後に即修正した。

## TypeScript ルールを追加した（`as` と `any` の禁止）

### 背景

プロジェクトに TypeScript 固有のルールファイルがなかった。`as` による型アサーションと `any` 型は型安全性を損なうため、原則禁止・使用時はコメント必須というルールを明文化した。

### 変更内容

`.github/rules/typescript.rules.md` を新規作成した。

**ルール概要：**

- `as`（型アサーション）は禁止。使うときは直前の行に理由コメントを付ける
- `any` 型は禁止。使うときは直前の行に理由コメントを付ける
- 代替手段として型ガード・`unknown` 型の使用を推奨

### 設計上のポイント

テストをレイヤーに沿って分割することで、失敗箇所がすぐに特定できるようになった。
