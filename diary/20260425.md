仕様書（`docs/design/backend/api.md`、`docs/design/db/db.md`）はすでに揃っていたので、それを TDD サイクルで実装することにした。

### Red フェーズ：テストを先に書く

`backend/src/app.test.ts` を新規作成した。

Hono の `app.request()` を使えばサーバーを起動せずに HTTP レベルのテストが書けるので、それを活用した。  
各テストが独立して動くように、`clearStorage()` を実装側からエクスポートして `beforeEach` でストアをリセットする設計にした。

カバレッジの設計方針：

- 正常系（201 / 200 のレスポンス形状チェック）
- バリデーションエラー（422：欠如・空文字・空白のみ・文字数超過）
- 一意制約違反（409：重複 name）
- 存在しないリソース（404）
- ソフトデリート後に一覧・単体取得ともに見えなくなること
- App 削除時に配下の Todo もカスケード削除されること
- 境界値（name が100文字 → OK、101文字 → 422）

この段階では実装がないため全テストが失敗する（Red）。

---

### Green フェーズ：テストを通す実装

`backend/src/index.ts` を全面的に書き直した。

**設計のポイント：**

- ストレージは `Map<string, AppRecord>` / `Map<string, TodoRecord>` のインメモリ実装（DB 接続なし）
- `deletedAt` は内部レコードに持つが、レスポンス（DTO）には含めない
- ソフトデリートは `deletedAt` にタイムスタンプをセットするだけ
- App 削除時のカスケードは `todosStore` をループして同じタイムスタンプをセット
- バリデーション関数（`validateName` / `validateTitle`）は失敗時にメッセージ文字列を返し、成功時に `null` を返す設計にした

### Review フェーズ：コードレビューで2件のバグが見つかった

レビュー結果は `review/todo-api-20260425.md` に保存した。

#### バグ1（P2）：`Boolean()` 強制変換によるサイレントなデータ破壊

PUT Todo の `completed` フィールドで `Boolean(body.completed)` を使っていた。  
`"false"` という文字列を送ると `true` に変換されてしまう。

```typescript
// 修正前
t.completed = Boolean(body.completed)

// 修正後
if (typeof body.completed !== 'boolean') {
  return c.json(errorRes('VALIDATION_ERROR', 'completed must be a boolean'), 422)
}
t.completed = body.completed
```

#### バグ2（P3）：ホワイトスペースのトリムが保存まで一貫していなかった

バリデーションはトリム後の文字列で行うのに、保存はトリム前の元の値でしていた。  
`"  My App  "` と `"My App"` が別名として共存できてしまうため、一意制約が形骸化していた。

```typescript
// 修正後：保存前に trim()
const name = (body.name as string).trim()
```

両バグともレビュー後に即修正した。

### 残課題・注意点

- **テストは実際には未実行**：環境に PowerShell 7+ がないため `npx vitest run` を走らせられなかった。静的解析で正確性を確認済みだが、実際にローカルで `cd backend && npx vitest run` を実行して全テストが通ることを確認したい。
- **カスケードデリートのテストに弱さあり**：レビューで指摘された通り、現在の cascade テストは App と Todo の ID が一致しないことによる 404 を検証しており、cascade ロジック自体の正しさを直接証明できていない。将来的に store inspection ヘルパーを追加するか、テストの設計を見直すと良い。
- **実際の DB 接続は未実装**：今回はインメモリストアで代替しているため、MySQL への接続・マイグレーション・ORM 層は別途実装が必要。
